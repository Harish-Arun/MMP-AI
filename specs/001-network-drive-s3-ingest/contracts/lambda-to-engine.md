# Contract: Lambda → mmp-ai-engine Notification

**Branch**: `001-network-drive-s3-ingest` | **Date**: 2026-03-10  
**Source**: FR-013, FR-005  
**Consumer**: `mmp-ai-engine` (feature 004: api-and-agents)  
**Producer**: `lambda/handler.py` (S3 event trigger)

---

## Overview

When a file is successfully uploaded to S3 by the ingestion service, S3 automatically invokes the Lambda function. The Lambda function sends a single HTTP POST request to the mmp-ai-engine REST API to trigger document processing.

```
SFTP Server
    │
    │  (ingestion-service polls, uploads)
    ▼
S3 Bucket  ──[ObjectCreated event]──►  Lambda (mmp-ai-s3-trigger)
                                              │
                                              │  HTTP POST
                                              ▼
                                       mmp-ai-engine
                                       POST /api/v1/workflows/trigger
```

---

## Request

### Endpoint

```
POST /api/v1/workflows/trigger
```

The engine URL is configured via the `ENGINE_REST_URL` environment variable on the Lambda function.

### Headers

```
Content-Type: application/json
X-Source: s3-lambda-trigger
X-Request-Id: <UUID generated by Lambda per invocation>
```

### Body (JSON)

```json
{
  "s3_bucket": "mmp-ai-documents",
  "s3_key": "ingest/payment_instruction_001.pdf",
  "filename": "payment_instruction_001.pdf",
  "file_size_bytes": 204800,
  "detection_timestamp": "2026-03-10T09:15:00Z",
  "sha256_hash": "e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855"
}
```

### Field Definitions

| Field | Type | Required | Description |
|---|---|---|---|
| `s3_bucket` | `string` | ✅ | S3 bucket name containing the uploaded file |
| `s3_key` | `string` | ✅ | Full S3 object key including prefix (e.g. `ingest/filename.pdf`) |
| `filename` | `string` | ✅ | Bare filename without prefix — the unique document identifier |
| `file_size_bytes` | `integer` | ✅ | File size in bytes at upload time |
| `detection_timestamp` | `string` (ISO 8601 UTC) | ✅ | When the file was first detected on the SFTP server |
| `sha256_hash` | `string` | ✅ | SHA-256 hex digest — engine uses this for integrity verification |

---

## Response

### Success

The engine MUST return **HTTP 202 Accepted** to acknowledge receipt of the trigger. The workflow is initiated asynchronously; a `202` does not mean the workflow has completed, only that it has been accepted and queued.

```
HTTP/1.1 202 Accepted
Content-Type: application/json

{
  "workflow_id": "wf_abc123",
  "status": "accepted",
  "message": "Workflow triggered for payment_instruction_001.pdf"
}
```

### Error Responses

| Status | Meaning | Lambda Behaviour |
|---|---|---|
| `202 Accepted` | Trigger accepted | ✅ Lambda exits successfully |
| `400 Bad Request` | Invalid payload | Lambda logs ERROR, **does not retry** (non-retryable — payload is malformed) |
| `409 Conflict` | Duplicate trigger (engine already processing this file) | Lambda logs WARNING, **does not retry** (idempotency — engine deduplicates) |
| `5xx` | Engine error / unavailable | Lambda retries with exponential backoff up to `MAX_RETRIES` (default: 3), then raises → invocation fails → Lambda DLQ captures event |
| `timeout` (>10s) | Engine unresponsive | Treated as 5xx, same retry behaviour |

---

## Retry and Reliability

### Lambda Retry Configuration

```
MAX_RETRIES = 3
BACKOFF_MULTIPLIER = 2  (seconds: 1s → 2s → 4s)
REQUEST_TIMEOUT_S = 10
```

Configured via environment variables on the Lambda function:
- `ENGINE_REST_URL` — mmp-ai-engine base URL
- `MAX_RETRIES` — default `3`
- `REQUEST_TIMEOUT_S` — default `10`

### Dead Letter Queue (DLQ)

A DLQ is configured on the Lambda function. Events where all retries are exhausted (persistent 5xx or timeout) are sent to the DLQ for manual inspection and replay. This guarantees no notification is silently lost (FR-007).

DLQ message retains the original S3 event; operations can replay via Lambda console or CLI.

---

## Idempotency

The mmp-ai-engine MUST handle duplicate `POST /api/v1/workflows/trigger` calls for the same `filename` gracefully:
- If a workflow for `filename` is already in progress → return `409 Conflict`
- If a workflow for `filename` has already completed → return `409 Conflict` (do not re-trigger)

The ingestion service's deduplication (FR-004, `UploadRecord` by `filename`) prevents double uploads under normal operation, but the engine-side `409` guard handles any edge cases (e.g., DLQ replay of an already-processed file).

---

## LocalStack Wiring

For local development, the Lambda is deployed to LocalStack and configured with:

```bash
ENGINE_REST_URL=http://host.docker.internal:8000  # or Docker network service name
```

The S3 bucket notification → Lambda trigger is identical in LocalStack and AWS. No code changes required between environments — only the `ENGINE_REST_URL` and `s3_endpoint_url` env vars change.

## Example Lambda Invocation (S3 Event)

The Lambda receives a standard AWS S3 event payload. The handler extracts the necessary fields:

```json
{
  "Records": [{
    "s3": {
      "bucket": { "name": "mmp-ai-documents" },
      "object": {
        "key": "ingest/payment_instruction_001.pdf",
        "size": 204800,
        "eTag": "abc123"
      }
    }
  }]
}
```

The Lambda reconstructs `filename` by stripping the `s3_key_prefix` from `key`. The `sha256_hash` and `detection_timestamp` are retrieved from the S3 object metadata (set by `s3_uploader.py` at upload time via `Metadata` header).
